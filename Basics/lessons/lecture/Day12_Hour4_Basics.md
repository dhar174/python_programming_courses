# Session 12, Hour 4: Final Assessment and Certification-Style Review Facilitation Script

## Purpose of this hour in program operations

This script is the final assessment facilitation guide for Hour 48 of the Basics course. This is not a standard coding lesson. This is the closing checkpoint of the program where learners demonstrate end-to-end competency, complete a certification-style conceptual and code-reading review, receive individualized feedback, and leave with explicit next-step study targets.

The operational model for this hour is intentionally different from a typical one-hour lesson segment. Hour 48 is delivered as an extended final-assessment block that runs for 60 to 75 minutes in program operations. Inside this block, the instructor still completes the required talk points and live demo, then transitions directly into assessment activities without introducing any new topics. This structure resolves timing ambiguity by treating the full 60 to 75 minute period as one integrated assessment flow rather than a normal split lecture and lab hour.

Use this script near verbatim. You can adapt tone to your class culture, but do not remove any required evidence checkpoints. Every learner must produce assessable evidence and every learner must receive specific feedback before dismissal.

## Required outcomes for Hour 48

By the end of this final block, each learner must show two outcomes explicitly:

- Outcome 1: End-to-end competency demonstration in Basics objectives.
- Outcome 2: Next-step study targets documented for post-course practice.

Instructor speaking line:

“Our goal now is simple and important. First, you will show that you can build and explain a complete Basics-level Python solution from input to output with persistence and error handling. Second, you will leave today with a personalized practice target so you know exactly what to sharpen next.”

Evidence expectations for outcomes:

- Outcome 1 evidence is captured through capstone demonstration scoring and explanation checks.
- Outcome 2 evidence is captured through a per-learner target artifact listing one verified strength and one specific next-practice focus.

If a learner has strong coding output but weak explanation quality, do not mark full completion yet. The course expectation includes both implementation and explanation ability at a high level.

## Scope guardrails for this final block

Stay strictly within Basics scope. This final session validates what learners already studied. Do not teach new frameworks, new libraries, or advanced architecture patterns. Keep all tasks in familiar territory:

- Types and conversions
- Strings and formatting
- Data structures
- Conditionals and loops
- Functions and modules
- Basic class usage
- File handling with text and JSON
- Path handling with pathlib
- Exception handling for common Basics cases

Instructor speaking line:

“Nothing in this block should feel like a surprise topic. We are not adding new content. We are validating that your existing toolkit is reliable and that you can reason about code under assessment conditions.”

## Operational timing model with no arithmetic contradiction

Use this section exactly when setting class expectations. This directly addresses the timing contradiction concern.

Operational model statement:

- Hour 48 is delivered as an extended final-assessment block of 60 to 75 minutes.
- The 10 to 15 minute talk points and 5 to 10 minute live demo occur at the beginning of that same block and are part of the assessment flow setup.
- The final assessment activities continue immediately after, still within the same 60 to 75 minute operational block.

Recommended 75-minute timeline example:

- Minute 0 to 3: Opening, framing, scoring transparency.
- Minute 3 to 15: Instructor talk points and domains checklist framing.
- Minute 15 to 22: Live read-code-predict-output demo.
- Minute 22 to 55: Capstone demos with rubric scoring.
- Minute 55 to 70: Short written and code-reading review.
- Minute 70 to 75: Individual feedback statements and exit quick check.

Recommended 60-minute compression fallback:

- Minute 0 to 2: Opening and expectations.
- Minute 2 to 12: Talk points compressed to essentials.
- Minute 12 to 17: Live demo compressed to one core prompt.
- Minute 17 to 40: Capstone demos with strict timeboxing.
- Minute 40 to 55: Written and code-reading review covering all 10 prompts in rapid mode (15-minute minimum).
- Minute 55 to 60: Per-learner strength and next-practice target capture with verbal quick check.

Compression rules when only 60 minutes are available:

- Keep all required components.
- Reduce speaking depth, not component coverage.
- Enforce strict presenter timers.
- Use rapid whole-class response methods for quiz prompts.
- Capture at least one sentence of feedback per learner before dismissal.

Instructor speaking line:

“This is one integrated assessment block. We will still do the review talk and the live code-reading demo first, then move into demonstrations and scoring. If we are on the 60-minute version today, I will run tighter timers and faster transitions, but we will complete every required checkpoint.”

## Instructor talk points segment, target 10 to 15 minutes

This segment must include the complete domains checklist and exam-style conceptual and code-reading pattern framing.

Use this structure:

1) Reconfirm what is in scope.
2) Walk domains checklist quickly but clearly.
3) Explain exam-style reasoning patterns.
4) Set confidence and strategy message.

Suggested speaking script:

“Let’s anchor the full Basics domains checklist first. As I read each line, self-rate your confidence: strong, moderate, or needs review.”

“Domain one, types and conversions. You should be comfortable moving between strings and numeric types safely and understanding what happens if conversion fails.”

“Domain two, strings and formatting. You should be able to slice strings, inspect lengths, and format readable output with f-strings.”

“Domain three, data structures. You should confidently use lists, tuples, sets, and dictionaries for the right jobs and understand how indexing and key access can fail.”

“Domain four, conditionals and loops. You should be able to follow branch logic and loop boundaries, including where off-by-one errors happen.”

“Domain five, functions, modules, and basic class usage. You should understand function inputs and returns, imports between files, and simple class responsibilities.”

“Domain six, files and persistence. You should load and save text or JSON data, use pathlib for clean path handling, and list directories when needed.”

“Domain seven, exceptions. You should recognize and handle common errors such as ValueError, FileNotFoundError, and JSONDecodeError with useful fallback behavior.”

“Now for exam-style framing. Certification-style questions often test two skills at once: conceptual understanding and code reading. You may be asked to predict output, identify a bug source, choose the safest fix, or explain why one approach is more robust.”

“When reading assessment code, use this sequence: read variable state, trace control flow, track updates, then check output. Avoid guessing from pattern memory. Always follow values line by line.”

“When answering conceptual questions, use plain language first. If you can explain it simply, you probably understand it deeply.”

“Remember, reliable Basics skill means you can both write and read code. A lot of people can write snippets from memory. Fewer people can read unfamiliar code calmly and predict behavior. Today we are proving you can do both.”

Talk-point interaction prompts:

- “Which domain feels strongest for you right now, and why?”
- “Which domain would you review first tonight if you had 20 focused minutes?”
- “What is one common bug type you now catch faster than you could at the start of the course?”

Facilitation rule:

Do not deep-dive into new teaching. Keep momentum. If a learner asks a large topic question, acknowledge and park it for post-assessment office time.

## Live demo segment, target 5 to 10 minutes

This segment must be a quick read-code-predict-output exercise.

Instructor setup line:

“Let’s do one fast code-reading drill to warm up your brain for assessment mode. Do not run the code yet. Read first, predict second, then verify.”

Demo snippet option A, recommended:

    items = [1, 2, 3]
    alias = items
    alias.append(4)
    print(len(items))
    print(items[-1])

Ask learners:

- “What does the first print output?”
- “What does the second print output?”
- “Why does changing alias also change items?”

Reveal and explain:

- First output is 4.
- Second output is 4.
- Both names reference the same list object.

Optional second quick snippet if time allows:

    raw = "12"
    total = int(raw) + 3
    print(f"total={total}")

Prompt:

- “What happens if raw were ‘12a’ instead of ‘12’?”

Expected explanation:

- int conversion fails with ValueError, so validation or try and except is required.

Close the demo with this line:

“That is exactly the mindset for the next block: read precisely, predict before executing, explain the reason, then show a safe fix.”

## Final assessment block, part 1: capstone demo and rubric scoring

Target duration:

- 30 to 45 minutes in the 75-minute model.
- 28 to 30 minutes in the 60-minute compression model.

Presenter timebox per learner:

- Demo walkthrough: up to 3 minutes.
- Clarifying question and explanation check: up to 1 minute.
- Rubric mark and handoff: up to 30 seconds.

Enforcement line:

“At 3 minutes I will stop the demo to protect fairness for the whole class.”

Presentation order:

- Preassign order before starting.
- Call next presenter while current presenter finishes to reduce dead time.

Required capstone checklist items for pass evaluation:

1) Menu loop exists and allows repeated user interaction until exit choice.
2) Functions are split into modules with clear responsibilities.
3) At least one class is present and used meaningfully.
4) JSON persistence works across runs for save and load.
5) Exception handling is present for likely input or file failure points.
6) Path handling uses pathlib and avoids hardcoded absolute paths.
7) Program prompts and outputs are readable for a user.
8) Project structure is organized enough for another learner to follow.

Capstone rubric rows for scoring operations:

- Requirements met
- User experience
- Code organization
- Data persistence
- Demo and explanation
- Optional polish bonus for directory listing behavior

Rubric scoring scale suggestion:

- 0 = Missing or incorrect.
- 1 = Partial with major gaps.
- 2 = Mostly correct with minor issues.
- 3 = Fully meets expectation for Basics level.

Decision rule for pass or retry:

- Required checklist rule: All eight required checklist items must be demonstrated as present and functional.
- Rubric threshold rule: Learner must score at least 12 out of 15 across the five required rubric rows, with no required row below 2.
- Explanation-quality rule: Learner must meet at least four of five explanation indicators listed below.

If a learner fails either the checklist rule or rubric threshold or explanation rule, mark “Not yet complete” and assign targeted re-demonstration action.

Explanation-quality indicators, measurable:

1) Learner can state program purpose and user flow in under 45 seconds.
2) Learner can identify where key functions and class responsibilities live.
3) Learner can explain how data is saved and loaded between runs.
4) Learner can name at least one handled exception and fallback behavior.
5) Learner can describe one design choice tradeoff using plain language.

Instructor scoring prompt:

“In one minute, explain your architecture at a high level: entry point, where your core logic lives, how state is persisted, and where you handle failures.”

Moderation guidance for fairness:

- Ask equivalent depth questions to each learner.
- Do not coach through missing requirements during scoring.
- If demo breaks, allow one quick restart only if failure is environmental.
- Keep language neutral: describe evidence, not personality.

Evidence capture template for part 1:

Per learner record these fields:

- Learner name
- Demo start and end time
- Checklist items 1 through 8 marked yes or no
- Rubric scores for required rows
- Explanation indicators 1 through 5 marked yes or no
- Preliminary decision: pass now or retry with target

Suggested feedback phrasing during scoring:

- “Evidence confirmed for persistence across runs.”
- “Exception handling present for missing file case.”
- “Need clearer module separation before full completion.”
- “High-level explanation is clear and concise.”

## Final assessment block, part 2: short written and quiz review, target 15 to 20 minutes

This part must include all 10 code-reading prompts explicitly and clearly. Keep scope in Basics only.

Delivery methods:

- Whole class rapid-response with volunteer reasoning.
- Individual paper or digital short answer.
- Pair first, then cold-call explanation.

If time is tight, still state all 10 prompts verbally or visually, then collect concise answers.

Code-reading set with explicit prompts and instructor intent:

Prompt 1, predict output, string slicing plus len:

    text = "python"
    print(text[1:4])
    print(len(text))

Ask:

- “What are the two outputs in order?”
- “Why does slicing stop before index 4?”

Prompt 2, predict output, f-string with numeric conversion:

    amount = "7"
    bonus = 2
    total = int(amount) + bonus
    print(f"total={total}")

Ask:

- “What prints?”
- “Which line would fail if amount were ‘seven’?”

Prompt 3, find bug, off-by-one loop with range:

    values = [10, 20, 30]
    for i in range(len(values) + 1):
        print(values[i])

Ask:

- “Where is the bug?”
- “What is the safest correction?”

Prompt 4, find bug, list index error and guard with len:

    names = ["Ana", "Bo"]
    index = 2
    print(names[index])

Ask:

- “Why does this crash?”
- “Show one guard condition before access.”

Prompt 5, find bug, dict key access with in or get:

    scores = {"math": 90}
    print(scores["science"])

Ask:

- “What exception appears?”
- “Rewrite with a safe key check.”

Prompt 6, trace function return versus print:

    def double(x):
        print(x * 2)

    result = double(4)
    print(result)

Ask:

- “What appears in output and why?”
- “How should function change if caller needs returned value?”

Prompt 7, trace import from utils.py with working directory reminder:

Main file:

    from utils import greet
    greet("Sam")

utils.py:

    def greet(name):
        print(f"Hello, {name}")

Ask:

- “What must be true about file location for this import to work?”
- “What common naming mistake could break this import?”

Prompt 8, trace JSON load with missing file and FileNotFoundError handling:

    import json
    from pathlib import Path

    path = Path("data") / "state.json"
    with path.open("r", encoding="utf-8") as file:
        data = json.load(file)

Ask:

- “What happens if state.json is missing?”
- “Which exception should be handled?”

Prompt 9, trace JSONDecodeError recovery to defaults:

    import json

    raw = "{bad json}"
    data = json.loads(raw)
    print(data)

Ask:

- “What exception occurs?”
- “What is an acceptable Basics-level fallback strategy?”

Prompt 10, explain why pathlib avoids hardcoded paths:

Scenario statement:

“A script uses a hardcoded absolute Windows path in one machine and fails on another machine. Explain why pathlib with relative project paths is safer.”

Ask:

- “What portability problem is avoided?”
- “How does this support reliable collaboration?”

Instructor key points for debrief after each prompt:

- Emphasize reasoning, not memorization.
- Ask at least one learner to justify line by line.
- Keep explanations concise to preserve schedule.

Scoring guidance for part 2:

- Suggested target: 7 out of 10 correct for strong readiness.
- If below 7, assign focused review plan before exit.
- Track conceptual misses by domain to personalize feedback.

## Completion criteria with measurable pass and fail definitions

Completion is binary at this final checkpoint: complete now or not yet complete.

Pass definition:

A learner is marked complete only when all of the following are true:

- Capstone required checklist items 1 through 8 are all met.
- Rubric threshold is met at 12 out of 15 or higher across required rows, with no required row below 2.
- Explanation-quality indicators meet at least 4 out of 5.
- Learner participates in short written and code-reading review and provides responses to all prompt areas.
- Per-learner artifact is recorded with one strength and one next-practice target.

Fail definition, not yet complete:

A learner is marked not yet complete if any one of the following is true:

- One or more required checklist items are missing.
- Rubric total is below threshold or a required row score is below 2.
- Explanation-quality indicators are fewer than 4 of 5.
- Learner cannot explain code at a high level when prompted.

High-level explanation requirement clarification:

“At a high level” means the learner can describe what the program does, how data moves, where major logic lives, and how errors are handled, without needing line-by-line recitation of every statement.

Decision announcement script:

“Based on evidence from your demo, rubric, explanation check, and code-reading review, your status is complete now or not yet complete. If not yet complete, I will give you one focused remediation target and your reassessment path.”

## Rubric operations and moderation detail

Use these operational rules to keep scoring valid and defensible:

Consistency rules:

- Apply identical rubric definitions to every learner.
- Do not change thresholds mid-session.
- Record evidence at moment of observation.

Neutral language rules:

Interruption rules:

- If a learner exceeds time, stop politely.

- Confirm score based on demonstrated evidence only.

Documentation minimum:

- One explanation-quality record per learner.
- One strength plus next target artifact per learner.

## High-level explanation coaching lines

When learners struggle to explain architecture, use one neutral prompt:

“Walk me through your program from user input to saved output in four short steps.”

If still unclear:

“Point to where your core logic lives, where persistence happens, and where errors are handled.”

If learner over-focuses on line detail:

“Pause on line-by-line details for now. I need your high-level map first.”

Signs explanation quality is sufficient:

- Learner uses clear sequence language.
- Learner connects code sections to responsibilities.
- Learner names at least one protective decision.

Signs explanation quality is insufficient:

- Learner only reads code lines aloud without synthesis.
- Learner cannot locate persistence or error handling responsibilities.
- Learner cannot explain tradeoff or design rationale.

## Fairness, confidence, and classroom climate

Assessment rigor and learner confidence can coexist. Use clear, respectful language and predictable process. Learners perform better when expectations are visible and stable.

Confidence-support lines:

- “You are being assessed on what we practiced repeatedly.”
- “A calm explanation beats a rushed explanation.”
- “If you forget a term, describe behavior in plain language.”

Rigor lines:

- “I score based on observed evidence only.”
- “I cannot infer features that were not shown.”
- “Meeting threshold means meeting all required criteria.”

Balance strategy:

- Validate effort.
- Enforce criteria.
- Provide focused next action.

## Handling blockers during the final block

Common blocker 1, demo fails to launch:

- Allow one quick relaunch.
- If unresolved in one minute, move learner to end of queue.
- Continue with others to preserve class timeline.

Common blocker 2, learner panics during explanation:

- Ask one smaller prompt.
- Accept plain-language explanation.
- Score criteria based on evidence, not speaking style preferences.

Common blocker 3, time overrun pressure:

- Tighten transition language.
- Skip extended debrief.
- Preserve required three-part structure.

Common blocker 4, disagreement with prompt answer:

- Ask learner to trace values line by line.
- Compare against observed program behavior.
- Award partial credit when reasoning shows partial understanding.

## Rationale for this facilitation design

This design protects rigor, fairness, and clarity. Rigor is protected by explicit thresholds and required evidence. Fairness is protected by equal timing and consistent questions. Clarity is protected by visible outcomes, transparent scoring, and specific next steps. Learners leave knowing exactly where they stand and exactly what to do next.

This model also mirrors real-world expectations. In practical software contexts, developers must demonstrate not only code that works but also code they can explain and maintain. They must read code written by others, identify risk quickly, and apply safe corrections. That is why this final block emphasizes code-reading patterns, explanation quality, and measurable competency signals.
